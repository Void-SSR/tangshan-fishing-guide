### 2026-04-16 第六轮

- 本轮目标：修正 GitHub Pages 发布后首页空壳风险，并把“大南先验收再交付”固化成强制门槛
- 已完成：
  - 确认阶段性交付前必须先经过大南 QA Gate，并写入复用与交接协议
  - 识别出 GitHub Pages 下“不带尾斜杠访问”会让相对路径资源失效的兼容问题
  - 为所有页面增加动态 `<base>` 注入，兼容 `file://`、GitHub Pages 子路径、以及不带尾斜杠访问
  - 识别出 Service Worker 缓存名仅按日期生成，会导致同一天重复发布时命中旧壳缓存
  - 将 Service Worker 缓存名改为带秒级时间戳的构建标识
  - 顺序重跑 `npm run test:smoke` 并通过
- 已吸收结论：
  - GitHub Pages 成功部署不等于用户一定看到最新页面，路径兼容和缓存版本同样是发布门槛
  - 发布后出现“页面几乎空白”时，必须优先排查：路径基准、Service Worker 缓存、而不是先怀疑数据丢失
  - QA Gate 不能只看 workflow 变绿，必须把用户实际访问路径纳入验收
- 未解风险 / 下一步：
  - 需要把这轮修复推送到 GitHub，再触发 Pages 重建
  - 推送完成后，继续观察首页是否仍受旧 Service Worker 影响；必要时再加显式缓存更新提示
