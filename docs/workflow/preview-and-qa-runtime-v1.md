# 预览与 QA 运行机制 v1

## 目的

解决两个长期问题：

1. 当前聊天 / 沙箱环境无法稳定直接打开本地网站
2. QA 角色不能把“真实浏览器预览”当成唯一验收方式

## 事实边界

当前环境里已经确认存在以下限制：

- `file://` 不能作为浏览器自动化入口
- 本地端口绑定可能被沙箱拒绝，典型表现是 `EPERM`
- 浏览器自动化和系统预览链路都可能被权限拦截
- 因此“我这里打不开网站”不等于“网站本身没有内容”

## 预览链路优先级

在进入任何预览链路之前，先过本地安全底线：

- `npm run safety:preflight`
- 规则见 [local-safety-baseline-v1.md](/Users/ssr/Downloads/Openai%20Codex/tangshan-fishing-guide/docs/workflow/local-safety-baseline-v1.md:1)

### 1. 宿主机真实预览

优先入口：

- `site/preview-local.command`

作用：

- 先构建站点
- 在宿主机启动本地 HTTP 服务
- 尝试自动打开浏览器

这是最接近真实用户浏览体验的链路。

### 2. 预览环境诊断

诊断入口：

- `npm run preview:diagnose`

作用：

- 检查 `dist` 是否存在
- 检查 `node / npm / python3 / open` 是否可用
- 检查本地端口是否可以绑定

如果这一层失败，就不要继续强求当前环境做浏览器预览。

### 3. 浏览器自动化

环境允许时：

- 使用 Playwright / MCP 浏览器工具做真实浏览式回归

环境不允许时：

- 不把这一步作为当前回合的阻断前提

### 4. 静态 smoke

当前环境受限时的最低质量门槛：

- `npm run test:smoke`

静态 smoke 必须继续覆盖：

- 三层结构红线
- 危险提示规则
- 图片状态与文案一致性
- 详情页完整性
- 本地直接浏览所需的相对路径资源

## QA 角色降级规则

### 正常情况

- 大南拥有者执行真实浏览式验收

### 拥有者失效但可接任

- 新大南先读职责文档和日志
- 再继续测试

### 拥有者失效且不可接任

- 主线程临时代行大南职责
- 结果必须写回 `danan-log-v1.md`
- 后续接任者直接续接，不重新定义角色

## 当前建议

对于这个项目，默认采用“双链路”：

- 宿主机预览链路：看真实网站
- 沙箱静态验收链路：守结构和数据红线

这样可以避免把环境限制误判成产品故障。
