# 唐山钓鱼指南建站方案 v1

## 总原则

先把内容体系和页面骨架做对，再开始正式开发。

因为当前已经有独立信息库，后面建站最适合走“静态站 + 数据驱动”的路线，而不是直接做一个重后台。

## 我建议的技术路线

### 第一阶段：静态站快速落地

推荐技术组合：

- `Astro`
- 原生 `CSS`
- 本地 `JSON` 数据文件
- 少量浏览器端 `JavaScript` 做筛选和切换

推荐理由：

- 非常适合内容型网站
- 页面加载快
- SEO 友好
- 结构清晰
- 后续加 JSON、Markdown、脚本都很顺手
- 不会像大型前端框架那样把项目做重

### 第二阶段：内容导入半自动化

等钓点数量稳定增长后，再补：

- 标准化导入模板
- 数据校验脚本
- 更新日志自动生成
- 待复核内容自动标记

## 为什么我不建议直接继续用根目录那个旧站

原因很明确：

- 你已经明确要求不要动
- 旧站内容完全是别的项目
- 它的结构和这个钓鱼指南的内容模型并不匹配

所以后面真正开发时，建议在 `tangshan-fishing-guide` 下开一个新的站点子目录，例如：

- `tangshan-fishing-guide/site`

## 数据组织建议

建议后面保持三层：

### 1. 原始来源层

- 用户提供资料
- 外部补充资料
- 后续修订原文

### 2. 结构化内容层

- 钓点数据
- 目标鱼数据
- 季节与潮汐建议
- 合规与安全提醒

### 3. 前端展示层

- 首页模块
- 列表页模块
- 详情页模块
- 更新页模块

## UI 方向建议

我建议的方向不是“渔具电商风”，也不是“资讯门户风”。

更适合走这条线：

- 视觉关键词：海港、潮汐、堤岸、风、钢铁海岸、实用导航
- 色彩关键词：深海蓝、雾灰蓝、沙色、锈橙小面积点缀
- 体验关键词：稳、清楚、可信、可快速决策

页面气质建议：

- 首页有氛围
- 列表页高效率
- 详情页像一份可执行的出钓卡片

## 设备适配原则

这个项目按“手机 App 化网页”来设计，优先适配 iPhone 竖屏。

建议把常见 iPhone 逻辑宽度作为主基准：

- `390pt`
- `393pt`
- `402pt`

因此前端实现时需要优先考虑：

- 顶部安全区
- 底部安全区
- 单手操作触达
- 列表页卡片在窄屏下的阅读效率

桌面端后续再做扩展适配，但不作为第一优先级。

## 交互工具建议

真正开始制作时，我建议我们分三类工具用：

### 1. 内容与数据层

- `JSON`：主数据源
- Node 脚本：导入、清洗、生成索引

### 2. 设计与前端层

- `Astro`：页面结构
- 原生 `CSS`：视觉定制
- 少量前端脚本：筛选器、切换器、搜索

### 3. 后续增强层

- 地图 API：等经纬度补齐后再接
- 自动化脚本：等导入流程固定后再上
- 定时更新机制：等来源稳定后再考虑

## 信息同步系统建议

推荐从最稳的人工确认流开始：

1. 你发来新资料
2. 我先整理成标准字段
3. 你确认
4. 数据文件更新
5. 网站展示自动同步

这一步不复杂，但非常稳。

## 图片与真实性校验策略

用户已经允许联网核验与找图，因此后续制作时建议同步进行两类工作：

### 1. 真实性核验

- 优先核验合规信息、休渔信息、港区管控、海洋牧场身份
- 对“能不能进”“是否合规”“是否仍开放”这种动态信息单独标记复核状态

### 2. 图片素材策略

- 优先使用官方景区、官方报道、政府/机构发布图
- 其次使用能明确对应地点的公开环境图
- 工业港区和防波堤点位若缺少高质量公开图，先用区域图或环境图占位

图片抓取与落地时要同时记录：

- 图片来源
- 图片对应点位
- 是否可替换
- 更新时间

## 新信息导入机制建议

后续可以固定一个导入模板，字段与当前信息库一致。

建议新资料每次都保留：

- 原文
- 来源时间
- 来源类型
- 是否已确认
- 是否需要复核

## 自动更新系统建议

我建议分三档推进：

### A 档：手动更新

- 最稳
- 最适合前期

### B 档：半自动更新

- 新资料放进指定目录
- 脚本转换成标准 JSON
- 自动生成更新记录

### C 档：自动巡检

- 标记长时间未复核的点
- 标记规则类内容的复核提醒
- 自动生成待确认列表

## 现在就可以进入开发前准备的内容

如果你认同这条路线，下一步我建议直接做两件事：

1. 把 `knowledge-base.json` 再拆成更适合前端读取的文件结构
2. 开始画首页和列表页的真实页面草图

这两步一做完，我们就可以进入正式制作阶段。
